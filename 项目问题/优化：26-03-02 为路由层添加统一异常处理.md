### 一、FastAPI提供的异常处理机制
在FastAPI类中有一个装饰器函数`exception_handler`，它会在内部调用其父类Starlette的`add_exception_handler`，将装饰器修饰的具体函数存放在Starlette的字典类型属性`exception_handlers`中，键为异常码，值为函数对象：
![[Pasted image 20260403144209.png]]
其中`decorator`是一个闭包函数，当它被返回后，它接收了外函数的信息`self`和`exc_class_or_status_code`后，作为闭包对象另外存在堆区中，等到调用`decorator(func)`时仍然可以将函数绑定到正确的FastAPI应用上面。
![[Pasted image 20260403144254.png]]
![[Pasted image 20260403144328.png]]
FastAPI会在应用处理请求的过程中抛出异常时，从`exception_handlers`字典中查找对应异常码的执行器，找到的话框架就会执行被该装饰器的函数，该函数的返回值（通常是一个JSONResponse）将直接作为HTTP响应返回给客户端，而不会让异常继续向上传播导致默认的错误页面或500响应。源码中该装饰器给的示例为：
```python
from fastapi import FastAPI, Request
from fastapi.responses import JSONResponse

class UnicornException(Exception):
	def __init__(self, name: str):
	    self.name = name

app = FastAPI()

@app.exception_handler(UnicornException)
async def unicorn_exception_handler(request: Request, exc: UnicornException):
	return JSONResponse(
		status_code=418,
		content={"message": f"Oops! {exc.name} did something. There goes a rainbow..."},
	)
```

### 二、实现应用的统一异常处理
编写`register_exception_handlers()`函数，为应用注册若干异常处理器，每个异常处理器都会返回一串格式相同的JSON格式的数据，包含三个字段：`code`、`message`、`detail`。这样的统一处理方便前端适配，且能够隐藏内部异常细节提高安全性。
```python
"""注册 FastAPI 全局异常处理器，统一 API 错误响应结构。"""
from fastapi import FastAPI, Request, status
from fastapi.exceptions import RequestValidationError
from fastapi.responses import JSONResponse
from starlette.exceptions import HTTPException as StarletteHTTPException
from app.exceptions import AppException

def register_exception_handlers(app: FastAPI) -> None:
    """将本模块内定义的处理器挂到应用实例上。"""
    @app.exception_handler(AppException)
    async def app_exception_handler(
        _request: Request,
        exc: AppException,
    ) -> JSONResponse:
        return JSONResponse(
            status_code=exc.status_code,
            content={
                "code": exc.code,
                "message": exc.message,
                "detail": exc.detail,
            },
        )

    @app.exception_handler(StarletteHTTPException)
    async def http_exception_handler(
        _request: Request,
        exc: StarletteHTTPException,
    ) -> JSONResponse:
        return JSONResponse(
            status_code=exc.status_code,
            content={
                "code": "http_error",
                "message": exc.detail if isinstance(exc.detail, str) else str(exc.detail),
                "detail": None if isinstance(exc.detail, str) else exc.detail,
            },
        )

    @app.exception_handler(RequestValidationError)
    async def validation_exception_handler(
        _request: Request,
        exc: RequestValidationError,
    ) -> JSONResponse:
        return JSONResponse(
           status_code=status.HTTP_422_UNPROCESSABLE_ENTITY,
           content={
                "code": "validation_error",
                "message": "请求参数校验失败",
                "detail": exc.errors(),
            },
        )

    @app.exception_handler(Exception)
    async def unhandled_exception_handler(
        _request: Request,
        _exc: Exception,
    ) -> JSONResponse:
        # 不向客户端泄露内部异常信息
        return JSONResponse(
          status_code=status.HTTP_500_INTERNAL_SERVER_ERROR,
          content={
                "code": "internal_error",
                "message": "服务器内部错误",
                "detail": None,
            },
        )
        
__all__ = ["register_exception_handlers"]
```
### 三、控制反转
**控制反转（Inversion of Control, IoC）** 是一种设计原则，它将程序流程的**控制权**从程序员编写的业务代码转移到**框架或容器**。它的核心思想：**好莱坞原则**——“不要打电话给我们，我们会打给你。”（Don‘t call us, we'll call you.）
#### 3.1 观察者模式
**异常处理器本质上就是观察者模式**，在LangChain的回调机制中事件是`on_llm_start`、`on_chain_error`等，在logging库中是Logger将日志事件分发给各种LoggingHandler。而异常处理器的事件就是不同类型的异常，如`AppException`、`HTTPException`、`ValidationError` 等。观察者就是通过`@app.exception_handler` 注册的异步函数，**当异常发生时，框架自动调用对应的观察者函数**。它关注的问题是**状态变更后自动广播**
#### 3.2 钩子
框架在特定流程（如请求处理、生命周期节点）中**预留的扩展点，开发者将自定义代码“挂载”到这些点上，框架执行到该点时自动调用**，它的典型命名即为`on_*`、`before_*`、`after_*`等。比如LangChain的`on_llm_start`和`on_chain_error`，FastAPI的中间件。它关注的问题是**在流程中插入自定义逻辑**，比如框架负责主流程，用户通过钩子添加日志、缓存、校验等横切关注点
#### 3.3 回调
回调是一种编程模式：**开发者将一个函数（称为回调函数）作为参数传递给另一个函数（或系统），后者在某个特定时机（例如操作完成、事件发生）调用这个函数**。接收方是接收参数的函数，调用时机由接收方确定，可以是同步或异步。
观察者模式和钩子都是对回调的一种应用，只是侧重点不同：**钩子是回调的一个直接应用（工具范畴），而观察者模式则是对回调的一种结构化、有状态的管理方式（设计模式范畴）**。正式因为钩子更像是一个工具，所以它能够**和不同设计模式实现配合**：
* 两者配合使用就是像LangChain的回调机制，或是FastAPI的异常处理一样，开发者为框架提供自己的逻辑，由框架发现事件并触发。
* 如果钩子和其他模式配合，比如像FastAPI的中间件，采用责任链模式，作用于应用路由层的外围，无论如何都会触发。**同样都是开发者自定义逻辑插入到框架的执行流程中，只不过因为和观察者模式配合，回调机制和异常处理就是“事件驱动”的；而与责任链模式配合的中间件，就是让请求按照责任链传递，无论什么情况中间件都会被触发**。钩子在流程中插入自定义逻辑的作用在这些场景下都没有变化，**只是“事件触发”的时机，被不同的设计模式影响了**。
* 至于日志系统的`Logger`（假设实例为`logger`），`logger`通过**手动调用`info()`、`error()`等方法，主动引起“事件触发”**，然后`logging`模块完成事件分发，而这些`Handler`、`Formatter`，都可以自定义。它同样也是观察者模式+钩子，但是之所以日志系统会需要`logger.info()`这样的**手动触发，是因为它并非框架，框架的一个作用就是让一部分的逻辑自动化地执行，而无需手动地安排触发时机**，这就是“控制反转”思想的体现：调用的控制权从开发者转移到框架
#### 3.4 依赖注入
**依赖注入（DI）是控制反转（IoC）的一种实现方式**，比如在uvicorn中，它调用`await app(scope, receive, send)`时，传入了自己实现的`receive`和`send`函数对象，外部实现，应用的代码/函数内部主动调用，编写函数的人无需考虑这些功能的实施，**只需要“声明”自己需要什么，这是依赖注入的思想**。**它和回调方向相反：依赖注入是应用主动调用；而回调是框架/外部主动调用**。
