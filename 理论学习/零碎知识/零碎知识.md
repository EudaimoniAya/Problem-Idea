## 模块的惰性加载与实例化位置
### 惰性加载
惰性加载指在真正需要的时候才创建对象或加载资源。它的使用场景包括：
1. **资源消耗大**：如果对象的创建需要消耗大量资源（内存、CPU、IO等），并且可能不会立即使用，那么惰性加载可以推迟这些消耗，提高启动性能。
2. **不确定性**：如果对象可能在整个程序运行中都不会被使用，那么惰性加载可以避免不必要的资源占用。
3. **循环依赖**：有时模块之间可能存在循环依赖，惰性加载可以帮助打破这种循环。
4. **动态配置**：如果对象的创建依赖于运行时才能确定的配置，那么惰性加载是必要的。
在运维层面，如配置和日志，通常希望在程序启动时就初始化，因为它们是基础服务，几乎所有的其他模块都会使用，而且初始化开销相对较小。所以，立即加载是合理的。
### 在`__init__.py`或模块内部创建实例 vs 调用方自行创建实例
这取决于模块的设计意图：
1. **在`__init__.py`或模块内部创建实例**：适用于整个模块只需要一个实例（单例），并且希望模块的使用者直接使用这个实例，而不关心其创建过程。例如，日志模块通常提供一个全局的logger实例。
2. **只在模块内部定义类，由调用方自行创建实例**：适用于需要多个不同配置的实例，或者实例的创建需要调用方提供参数的情况。例如，一个数据库连接池，不同的服务可能需要连接到不同的数据库，因此需要不同的实例。
### 单例模式的实现和暴露方式
单例模式通常用于确保一个类只有一个实例，并提供一个全局访问点。在Python中，可以通过模块级别的变量来实现单例（因为模块在第一次导入时会被缓存）。所以，在模块中创建一个实例并导出，就可以实现单例。
但是，**单例并不总是好的，它可能隐藏依赖关系，导致代码难以测试**。因此，需要权衡。
另一种做法是使用依赖注入，将单例实例作为参数传递给需要使用它的函数或类，这样在测试时可以替换为模拟对象。
**背后的工程思想：**
1. **单一职责原则**：一个模块或类应该只有一个引起变化的原因。将实例创建和使用分离，可以使模块更专注于其核心功能。
2. **依赖倒置原则**：高层模块不应该依赖低层模块，二者都应该依赖抽象。通过暴露接口（抽象基类）而不是具体类，可以降低耦合。
3. **控制反转**：将对象的创建责任从模块内部转移到外部（例如使用工厂模式或依赖注入容器），可以增加灵活性。
4. **可测试性**：如果模块内部直接创建实例，那么在测试时很难替换为模拟对象。通过将实例作为参数传递（依赖注入），可以更容易地进行测试。


## RAG
### Zero-shot思想
Zero-shot学习（Zero-shot Learning）是指在训练阶段不存在与测试阶段完全相同的类比，但是模型可以**使用训练过的知识来推广**到测试集中的新类别上。
这种能力被称为“**零样本**”学习，因为模型在训练时从未见过测试集中的新类别，在模型训练和提示词优化中均有体现。
**在模型训练中**：已知马（四脚兽）、虎（有条纹）、熊猫（黑白色）的特征，但是未训练过斑马的数据。此时告知模型斑马是四脚兽、有黑白色条纹，那么模型就可以在已知数据中推理，从而识别斑马。
**在提示词优化中**：Zero-shot思想用于**基于已训练的能力**，不提供任何示例，仅通过语言去描述任务的要求、目标和约束，**让模型直接生成结果**。即“用语言定义任务，解放（信任）模型的预训练知识”。
### Few-shot思想
Few-shot学习（Few-shot Learning）是指少样本学习，当模型在学习了一定类别的大量数据后，对于新的类别，只需要少量的样本就能快速学习，对应的有one-shot learning，单样本学习，也算样本少到为一的情况下的一种few-shot learning。
**在模型训练中（相似度判断方法）**：基于**少量企鹅样本**并结合相似度判断，推论未知图片内含“企鹅”
**在提示词优化中**：Few-shot主要用于基于少量示例，让模型参考示例回答。即”用示例定义任务，在模型的预训练知识的基础上，提升模型回答的对齐程度（如参考示例的格式）“

* SVN**集中式**版本控制系统：SVN服务器（Server）上维护了代码的所有版本。每个开发人员运行SVN客户端（Client），所有的版本控制都放在SVN服务器上，彼此通过网络通信（往往是内网）。

将公钥添加到添加到GitHub后，执行`git push/pull`时：
![[Pasted image 20260305064039.png]]
* 本地计算机将和远程GitHub进行协议版本交换（ssh1.0还是ssh2.0）、算法协商、DH密钥交换（双方创建临时密钥对），建立安全的加密通道（使用对称加密）。
* 然后进行**服务器身份验证**：GitHub用它的私钥签名某些信息，当`known_host`中已经有它的公钥时，就会自动验证；否则会提示用户确认
![[Pasted image 20260305051125.png]]
* 接着进行**客户端认证**：Git客户端用`id_rsa`私钥对服务器发送的**挑战**进行签名，GitHub用此前配置的用户公钥验证。
  其中“挑战”指一个真正持有私钥的人才能正确回应的“随机密码”。具体流程是：服务器生成一个随机数（如`123abc`），把这个随机数发送给客户端；然后客户端用自己的私钥对这个随机数进行签名，生成签名的结果假设为`S`，然后把`S`发回服务器；服务器用用户事先上传的公钥对签名`S`进行验证，看能否还原出原来的随机数`1123abc`。它被称为“挑战”是因为它就像是**服务器在看客户端是否能完成服务器发来的随机考验**。
* 认证成功后，所有传输的数据（代码、提交信息等）都通过之前协商好的会话密钥加密传输
【注】如果没有生成SSH密钥并在GitHub配置，则之前传输使用的就是HTTPS协议（HTTP + TLS/SSL），它们加密机制和SSH类似，所以传输的也是密文。但是因为**SSH配置了一次公钥后，后续`git push`就不用每次输入密码或令牌**（而当使用HTTPS传输时则无需配置SSH密钥，但需要输入GitHub的用户名和密码或个人访问令牌。**git保存环境变量只是用户不用输入，但是每次连接服务器时，认证仍然是必须的**）。或者一些企业内部自建的Git服务器可能只开放SSH端口。
另外，SSH负责的是**传输层的验证**（证明用户有权限推送到远程仓库），而Git提交记录中的用户名和邮箱是**内容层的元数据**，因为ssh虽然不用推送时提交密码/令牌，但是仍然需要提交作者信息（虽然假账号也可以，但是不应该这么做）
* 配置本地用户名：`git config --global user.name = 'xx'`
* 配置本地邮箱：`git config --global user.email = 'xx@email'`

使用`ssh -T git@github.com`测试能否通过GitHub配置的ssh公钥让远程仓库与本机客户端进行通信

### Git各阶段版本回退命令
* 工作区的代码改动不想要了（`git add`之前），可以通过`git checkout -- <文件名>`命令用本地仓库相应分支的代码覆盖掉已经修改但是不想要了的文件。另外如果没有`--`则表示切换到`<文件名>`的分支
* 暂存区的代码改动不想要了（`git commit`之前），可以通过`git reset HEAD [文件名]`，如果不加文件名则表示暂存区清空（但是工作区的改动仍然在）
* 本地仓库的**某一版本出错需要回滚，使用`git reset --hard <commit-hash>`**，其中`commit-hash`可通过`git log`命令查看
![[Pasted image 20260306023833.png]]
【注】HEAD是一个指向用户“当前所在位置”的指针，**是属于本地仓库的指针**，通常指向**当前分支的最新提交**。而`git reset HEAD [文件名]`能够清空暂存区的代码，是因为该指令将暂存区中指定的文件，**重置为HEAD所指向的版本（即当前最新提交）中的该文件内容**。另外`git reset --hard <commit-hash>`之所以可以回滚本地仓库，是因为HEAD把指针指向了对应`commit-hash`的仓库
*  `git reflog`：**默认显示HEAD指针的引用日志**，记录了导致HEAD变化的操作，如`checkout`、`commit`、`reset`、`merge`、`rebase`、`clone`等，但严格来说，**每个引用（分支、远程分支等）都有自己的 reflog**。例如 `git reflog main` 可以查看 `main` 分支的变更历史。
   `reflog`指令的输出格式：`<commit-hash> (<ref-names>) <refname>@{<n>}: <command>: <description>`

### Git的分支模型
在 Git 中，**分支本质上是一个指向某个提交对象的可变指针**。每次提交都会生成一个唯一的 SHA-1 哈希值，而分支名（如 `main`、`dev`）就是指向这个哈希值的“标签”。当新建一个分支时，**Git只是创建了一个新的指针**（HEAD是一个特殊的指针），并不会复制文件或目录，因此操作非常轻量。它的分支实现基于**提交对象**和**引用**：
* 每次提交会生成一个提交对象，包含作者、时间、提交信息、指向父提交的指针，以及一个指向树对象的指针（记录目录结构和文件内容）
* 分支就是一个指向某个提交对象的引用，存储在`.git/refs/heads`目录下
* 当切换分支时，Git会更新HEAD指向该分支，并更新工作目录的文件
* Git 的分支模型是**扁平的**，没有严格的“父子”层级关系。因为没有任何一个文件记录了“某个分支是从哪个分支出来的”这种父子关系，**各分支完全平等**。
  这代表，假如在`main`分支上执行`git checkout -b feature/rag`，git也只是创建了一个新的指针`feature/rag`，让它指向`main`当前指向的那个提交。**但git不会记录`feature/rag`是从`main`创建的这一关系，即如果`main`被删除，`feature/rag`依然可以独立存在**。
* 而分支中的`feature/`仅仅是分支名字的一部分，一般称为命名前缀，而不代表目录层级。如团队可以约定所有新功能分支以`feature/`开头，所有修复以`hotfix/`开头，但git本身不会把`feature/`解释为文件夹或命名空间
### Git分支版本控制命令（无冲突情况）
工作时应该多在分支上进行新功能的开发和测试，当功能开发测试稳定后再把分支上的代码合并到代码仓库中
* `git checkout <分支名>`：签出（checkout）分支，签出分支相当于**切换整个工作目录到另一条开发线**。不过该命令只能切换到**已存在的分支**，添加选项`-b`则能够**创建并切换**到一个新分支，相当于连续执行`git branch <分支名>`和`git checkout <分支名>`
* `git switch <分支名>`：git新版（2.23+）的分支切换命令，相当于`git checkout <分支名>`，但只做切换且只能切换到已有分支。添加选项`-c`则是能创建并切换，相当于`git checkout -b <分支名>`。官方更推荐用`switch`切换分支，用`git restore`恢复工作区文件，这样语义更清晰，但是`git checkout`仍然可用

### 远程分支管理（GitHub端）
在GitHub创建了新分支后，git端可在`git pull`拉取代码时看到新分支的显示。远程创建一个新分支后，本地也**尽量创建一个同名的分支**去追踪（`git checkout -b <本地分支名> <远程仓库名>:<远程分支名>`），查看分支追踪情况使用`git branch -vv`命令。另外如果要设置已经存在的本地分支去追踪某个远程分支，则切换到该分支后，用`git branch -u <远程仓库名>/<远程分支名>`。
真实开发中，对于主干分支`main`的提交需要慎重，要保证功能开发测试完成才行，所以一般都会新开分支和本地新分支建立映射，开发人员就在本地的新分支上开发。
### Git工作流实践
项目代码远程仓库的分支：
* `master`或`main`：主干分支
* `dev`：开发分支
* `release`：发布分支
个人开发分支：
* `feature`：特性分支
* `bugfix`：bug修复分支
* `hotfix`：热更新分支
一般的git工作流如下：
1. 创建个人本地开发分支：`git checkout -b feature/add_new_line origin/dev`（基于远程`dev`分支的最新状态创建本地分支`feature/add_new_line`，这样能够**隔离开发，避免直接修改`dev`分支，保证主分支稳定**）
2. 个人本地分支推送到远程分支：`git push origin feature/add_new_line:feature/add_new_line`（将本地`feature/add_new_line`分支推送到远程仓库，并在远程仓库自动创建/更新同名分支）
3. 提交个人远程代码分支和目标代码合入分支的MR（**Merge Request，合并请求**，GitLab术语，**GitHub称为PR**），请求将自己的`feature/add_new_line`分支合并到目标分支，如`dev`，里面会显示代码差异、提交历史、并允许团队讨论。**所以提交PR需要先开发完了一些东西后再提交，可以开发完功能提交（常见），也可以开发一部分提交（增量式开发，开发大功能时先分多个小PR逐步合并），但是要避免空PR。如果希望利用PR评论区进行早期设计讨论，需要在标题加`[WIP]`表示Work In Progress**。相关负责人进行CR（Code Review），即其他开发者阅读你的代码，提出改进建议或确认没有问题，审查通常在MR的讨论区进行
4. 相关负责人提出意见，本地修改相应的代码，推送到对应的远程代码分支上
5. 代码CR意见处理完，相关负责人进行代码`merge`，**代码修改从`feature/add_new_line`合入`dev`分支**，完成开发
6. 删除个人远程`feature/add_new_line`代码分支，确保远程仓库简洁，避免长期堆积无用分支
另外，用git在远程推送了一个新分支时，GitHub会检测到这个分支还没有被合并到默认分支（通常是`main`或`master`），并且该分支包含一些新的提交，它就会在仓库主页自动显示这个按钮，方便快速创建Pull Request（PR，即GitHub版的MR）。如果推送的是已有的PR分支（更新代码），则不会重复弹出，但是PR页面会自动更新。
![[Pasted image 20260307020032.png]]
在PR界面需要填写：
- **base 分支**：代码合并到哪里（通常是 `main` 或 `dev`）
- **compare 分支**：功能分支（默认已选好）
- **标题和描述**：说明这次改动的内容、目的，方便 reviewer 理解
- **Reviewers、Assignees、Labels 等**：可选，用于指定审查者、责任人、标签分类
填写后点击 “Create pull request”，就正式发起了 PR，后续可以在 PR 页面进行代码审查、讨论、持续集成检查等。
![[Pasted image 20260307020235.png]]
创建PR后需要等待审查者提出意见，根据反馈修改代码后再次推送更新（推送到同一个分支，PR会自动更新）。审查通过后，有权限的人会在PR页面点击“Merge pull request”完成合并，合并后GitHub通常会提示删除远程分支。
