# 一、Git底层机制
Git本质上是一个**内容寻址的文件系统**，上面封装了一个版本控制的用户和界面，当执行一个git命令时，它本质上是在操作**这个文件系统中的对象和引用**。

> [!问题] Linux自带ext4和XFS文件系统，Git是怎么用的？
> Git不是直接在用Linux的文件系统（File System，FS），而是**在Linux的物理FS之上实现的一套虚拟文件系统（Virtual File System，VFS）**。
> 文件系统存在**两个设计问题**：① **定义文件系统的用户接口**，它涉及定义文件及其属性、所允许的文件操作、如何组织文件的目录结构；② **创建算法和数据结构**，以便映射逻辑文件系统到物理外存设备。
> 那么既然Git是（虚拟）文件系统，它自然也会围绕这两大问题进行设计：
> * **定义用户接口**：Git重新定义了blob对象（文件）、tree对象（目录）、commit对象（文件元数据）、git指令（文件操作）、tree对象的嵌套（目录结构）、通过tree解析（路径访问）、分支标签HEAD（引用）等，用户只需通过Git命令这些接口来操作这些逻辑对象，而无需关系它们如何存储
> * **逻辑文件系统映射到磁盘物理块**：每个Git对象经过SHA-1哈希运算，得到一个40位的十六进制哈希值，它是对象在对象数据库的“地址”。Git将前两位作为目录名，后38位作为文件名，存储在`git/objects`下。最后这些文件就通过Linux的物理FS上映射到磁盘物理块上。
> 
> 作为VFS，它采用了面向对象的思想，抽象出了一个通用的文件系统模型，并且需要实现四种对象类型：超级块对象、索引节点对象、目录项对象、文件对象。但是**Git并没有完全照搬，而是为版本控制这个特定领域做了简化和重构**，比如Git是离线的单用户内容追踪系统，对象都是不可变、只读的，所以无需考虑并发、文件锁定、缓存一致性等问题。所以它**没有“文件对象”对应，因为无需维持“当前打开”的文件状态**。
# 二、Git的四种对象
Git的核心就是对象（VFS的面向对象思想），因此它维持了一个键值对数据库，向它存入内容它就会返回SHA-1处理后的哈希值。这些内容被存储为对象，一共有四种类型的对象：`blob`、`tree`、`commit`、`tag`。
## 2.1 Blob对象
用于存储文件的内容，但是它**不存储文件名**，只存储二进制数据。比如当执行`git add`时，Git会为每个文件的内容创建一个`blob`对象，并将其写入`.git/objects`目录（当`git add`和`git commit`创建的松散对象过多，累积到一定数量后，Git会将它们打包成更高效的`packfile`，存储在`.git/objects/pack/`目录下）
![[Pasted image 20260312070435.png]]
哈希值是40位，前两位被用来当作目录名，**后38位作为文件名**。
![[Pasted image 20260312070641.png]]
## 2.2 Tree对象
用于存储目录结构。它不直接存储文件内容，而是记录了当前目录下的文件`blob`和子目录`subtree`，以及它们的文件名（对应的Git对象键值）等信息。使用`git cat-file -p HEAD^{tree}`可以查看最新提交对应的顶层Tree：
![[Pasted image 20260312071238.png]]
打印内容可以看出，Tree对象很轻量级，每一行包括三部分信息：
* 文件模式：记录该条目的类型和权限，如`040000`是tree类型的子目录、`100644`是blob类型的普通文件、`100755`是blob类型的可执行文件等
* 类型和文件名：第二列和第四列的内容，其中类型由第一列推出，方便人类阅读
* 对象的SHA-1哈希：指向该条目在Git对象库的具体内容。如果是文件则指向Blob对象，如果是目录则指向Tree对象
有了以上信息，就可以去利用Tree对象构建出完整的目录结构（遍历树）。
## 2.3 Commit对象
用于指向一个tree对象（作为项目快照），它保存了指向的tree对象和它父commit对象的哈希值、作者、提交者、时间戳和提交信息。每当`git commit`命令执行，就会创建一个Commit对象，这个对象记录了本次提交的元数据：

使用`git cat-file -p <commit-hash>`查看提交对象的信息：
![[Pasted image 20260312124047.png]]
使用`git cat-file -p <tree-hash>`查看第一行树对象的信息：
![[Pasted image 20260312124305.png]]
从Commit对象存储的信息来看，它包含本次提交的文件快照（tree对象），以及它的`parent`字段指向前一个（线性历史）/多个（分支合并）提交，这些链最终构成一个有向无环图（DAG），即Git的提交历史。而`git log`则是沿着这些`parent`指针遍历commit对象，所以输出顺序是从新到旧。
## 2.4 Tag对象
类似于Commit对象，但是Tag对象指向一个特定的commit并附带标签信息，用于给历史打固定标记：`git tag -a v1.0 -m "version 1.0"`
## 2.5 四种对象共同实现版本控制【重要】
版本控制最大的挑战在于：如何在**还原项目**的前提下**节省空间**？
1. 最直接的想法就是**以项目为单位**分别存储，就像VMware的快照一样。但是这样难以协作和修改，而且更大的问题就是快照的存储占用（一张VMware一个G）。即使是以一个原文件为脉络，然后后续记录更改，大量减少存储空间占用。但未来还原文件时，仍需要从原文件开始，一级一级地修复，不适合细粒度的版本管理场景，所以Git并未采用这种方式
2. 所以很自然地**沿用共享内存的思路**，对于相同内容的文件指向同一位置。在这种设计下，项目的版本管理就是这样：按照版本存储，项目里全是目录，该有文件的地方全是指向文件的索引。但是又需要考虑同一文件的不同版本应该如何指向的问题，所以可以**引入哈希值实现内容的快速比对**，并且哈希值天然地适合作为索引，这样就有了Blob对象和Tree对象的雏形
3. 再进一步，目录（Tree）作为特殊文件，很适合和文件（Blob）共同管理，所以**文件和目录分别经过哈希处理**（SHA-1）后分散地存储在`.git/objects/`下面，**这40位的哈希值就类似VFS中的逻辑地址**。为了快速存取，将40位的哈希值前2位作为目录，后38位作为文件名（此时文件名不再参与查询）
4. **这样Blob对象和Tree对象的设计就已经成型**：给定一个项目根目录，Tree对象可以根据子目录和文件的哈希值查找对应的文件，从上到下遍历还原出整个项目。
5. 但是Blob对象和Tree对象分散地存储，**虽然结构清晰，但是仍然无法实现版本控制**，就像是散落在地上的蜘蛛网，内部结构清晰，但如果没有几个粘在墙上的点，那么再清晰的结构也无法体现。因为**要解决项目根目录在哪的问题**。针对该问题**Git设计了指向项目根目录的Commit对象**，它把蛛网的一角给提了起来，并且加上其中包含的父子关系，能够做到还原整个项目的演进脉络，且非常快地比对父子提交的不同。
6. 但是Commit随着不断增多，如果缺少进度标识，那么只在Commit信息中包含，仍然**不能方便标识里程碑**。因此就有了**Tag对象指向某一次提交，实现进度管理**。
7. 这样Blob、Tree、Commit、Tag四个对象就共同实现了版本管理。所有对象在存储层面都是平等的，都以文件形式存储在`.git/objects`目录下。
# 三、引用：分支、HEAD和标签
引用（reference）是Git的人性化指针，它们存储在`.git/refs`目录下，每个文件包含一个Commit哈希
## 3.1 分支
分支是一个指向某个Commit的可变指针，如`refs/heads/main`文件里存放了当前`main`分支的最新Commit哈希。
![[Pasted image 20260313050841.png]]
当执行`git commit`，Git就会创建新的Commit对象，然后将`refs/heads/<当前分支名>`更新为这个新Commit的哈希，如上图只有两个分支：`main`和`rr`，但是新创建分支提交后新文件中会有上一次Commit的哈希值，而在该分支新提交了改动后文件内容会变更为本次提交的哈希值，且其他分支不变：
![[Pasted image 20260313052011.png]]
![[Pasted image 20260313052508.png]]
## 3.2 HEAD
HEAD是一个特殊的引用，它通常指向当前分支而不是直接指向一个Commit，`.git/HEAD`文件的内容一般是`ref: ref/heads/main`，这表示当前处于`main`分支上。如果HEAD直接指向一个Commit，它的内容就是一个哈希值
## 3.3 标签
标签通常是一个固定的指针，不会移动。标签分为轻量标签和附注标签：
* 轻量标签只是一个引用文件，文件内容只是指向某个Commit的哈希，它不创建对象，无需SHA-1哈希处理，存放在`.git/refs/tags/`目录下面。
* 附注标签是一个Git对象，同时有一个指向它的引用。Tag对象需要经过SHA-1哈希处理，存储在`.git/objects`中，其引用存储在`.git/refs/tags/`目录下面
![[Pasted image 20260315101943.png]]
如图，第一条命令`git tag v1.0-lightweight`创建了一个轻量标签，它只是一个引用而非对象（**Git将所有对象以二进制形式存储**），所以使用Linux的`cat`命令就可以直接查看文件内容，**它的哈希值指向`test`分支的最新提交**。
第三条命令`git tag -a v1.0-annotated -m "release"`添加选项`-a`表示为附注标签，此时Git会创建一个Tag对象，以及一个**指向该对象的引用**。第四条命令是在查看创建附注标签的引用，它和轻量标签指向Commit对象不同，**它保存的哈希值指向的是Tag对象**。最后一条命令查看该哈希值对应的对象类型可看到是tag类型。
所以第五条命令使用`git cat-file`（Git提供的解析二进制文件命令）查看Tag对象，能够得到比轻量标签更多的信息。并且因为两次标签创建之间无其他提交，所以轻量标签和附注的标签对象指向的提交相同，都是`423527d83c119db01bad9aa15263a2ad3f111f6d`

# 四、`.git`目录结构
`.git`的核心目录结构为：
```
.git/
├── HEAD                 # 当前指向的引用
├── config               # 仓库配置
├── description          # 仅供 GitWeb 使用，描述仓库
├── index                # 暂存区（记录当前暂存的tree）
├── objects/             # 所有对象存储的地方
│   ├── 3b/              # 哈希前两位
│   │   └── 18e5...      # 哈希后38位，即一个对象
│   └── pack/            # 打包后的对象（节省空间）
├── refs/                # 引用（分支、标签等）
│   ├── heads/           # 本地分支
│   │   ├── main         # 文件内容为最新 commit 的哈希
│   │   └── test
│   ├── tags/            # 标签
│   └── remotes/         # 远程跟踪分支
│       └── origin/
│           ├── HEAD     # 通常指向 origin/main
│           └── main
├── logs/                # 引用日志（reflog）
│   ├── HEAD             # HEAD 的变动历史
│   └── refs/
│       ├── heads/
│       │   ├── main     
│       │   └── ...
│       └── remotes/
│           └── origin/
│               ├── main
│               └── ...
├── hooks/               # 客户端或服务端钩子脚本
│   ├── pre-commit.sample
│   ├── post-commit.sample
│   └── ...
├── packed-refs          # 打包后的引用（当引用过多时）
└── COMMIT_EDITMSG       # 最近一次提交的编辑信息（临时）
```
其中`.git/index`就是暂存区，它是一个二进制文件，记录了下一次`git commit`将生成的文件快照。可以用`git ls-files --stage`查看内容，输出格式为`文件类型:对象哈希:阶段号:文件路径`（阶段号通常为0，合并冲突时会出现1/2/3）：

# 五、命令对应
| 命令                                  | 底层操作                                                                                                                                                             |
| ----------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `git add <file>`                    | 1. 为文件内容计算 SHA‑1，创建 blob 对象并存入 `objects/`。  <br>2. 更新 `.git/index`，将文件路径与 blob 哈希关联起来。                                                                           |
| `git commit`                        | 1. 根据当前 index 创建 tree 对象（递归生成子目录的 tree）。  <br>2. 创建 commit 对象，包含 tree 哈希、父 commit 哈希、作者等信息，写入 `objects/`。  <br>3. 将当前分支引用（`refs/heads/<branch>`）更新为新 commit 的哈希。 |
| `git branch <name>`                 | 在 `refs/heads/` 下新建文件 `<name>`，内容写入当前 HEAD 指向的 commit 哈希。                                                                                                        |
| `git checkout <branch>`             | 1. 更新 HEAD 文件内容为 `ref: refs/heads/<branch>`。  <br>2. 读取目标 commit 的 tree，将其内容展开到工作目录，并同步更新 index。                                                                 |
| `git reset --hard <commit>`         | 1. 将当前分支引用指向 `<commit>`。  <br>2. 用该 commit 的 tree 覆盖 index 和工作目录。                                                                                                |
| `git reset --mixed <commit>`（默认）    | 1. 将当前分支引用指向 `<commit>`。  <br>2. 用该 commit 的 tree 覆盖 index，但工作目录不变。                                                                                              |
| `git reset --soft <commit>`         | 仅将当前分支引用指向 `<commit>`，不动 index 和工作目录。                                                                                                                            |
| `git merge <branch>`                | 1. 找到两个分支的共同祖先 commit。  <br>2. 生成合并后的 tree（三方合并），创建新的 commit 对象，其 parent 为两个分支的 commit 哈希。  <br>3. 将当前分支引用指向这个新 commit。                                          |
| `git rebase <upstream>`             | 1. 找到当前分支与上游分支的共同祖先。  <br>2. 将当前分支上从共同祖先之后的每个 commit 依次复制到上游分支之后（生成新的 commit 对象）。  <br>3. 将当前分支引用指向新生成的最后一个 commit。                                              |
| `git tag <tagname>`（轻量）             | 在 `refs/tags/` 下创建文件 `<tagname>`，内容写入指定 commit 的哈希。                                                                                                              |
| `git tag -a <tagname> -m <msg>`（附注） | 1. 创建 tag 对象，包含指向 commit 的哈希和标签信息，存入 `objects/`。  <br>2. 在 `refs/tags/` 下创建文件指向该 tag 对象的哈希。                                                                      |
| `git reflog`                        | 读取 `logs/HEAD`（或 `logs/refs/heads/<branch>`）中记录的引用变动历史。每次 HEAD 或分支移动（commit、reset、checkout 等）都会追加一条记录。                                                           |
| `git gc`                            | 将 `objects/` 中松散的对象打包为 `packfile`（放在 `objects/pack/`），同时压缩引用至 `packed-refs`，并清理不可达的悬空对象。                                                                         |
| `git fetch`                         | 从远程仓库下载缺失的对象，并更新 `refs/remotes/<remote>/` 下的远程跟踪分支。                                                                                                              |
| `git push`                          | 将本地分支引用的 commit 及其关联对象上传到远程仓库，并更新远程的引用。                                                                                                                          |
